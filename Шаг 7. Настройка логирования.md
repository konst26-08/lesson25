# Шаг 7. Настройка логирования

Документ описывает систему логирования проекта **StepUp Web MVP**: структурированные JSON-логи API, уровни **info / warn / error**, централизованное хранение в файле и инструменты анализа (включая промпты для AI).

---

## Цель

- **Структурировать логи** — каждая запись в формате JSON (JSONL: одна строка = одно событие).
- Добавить **уровни логирования**: `error`, `warn` (warning), `info`, `http`, `debug`.
- Настроить **централизованное хранение** — единый файл `logs/api.log` + stdout (docker / journald).
- Подготовить **промпты для поиска проблем** в логах с помощью AI.
- **Протестировать** типичные сценарии ошибок (unit + integration tests).

---

## Архитектура логирования

```text
HTTP-запрос
    │
    ▼
requestLogMiddleware ──► logger.http (JSON: method, path, status, durationMs)
    │
    ▼
Обработчик маршрута
    │
    ├─► Успех ──► (без лишнего шума, только access log)
    │
    └─► Ошибка ──► errorMiddleware ──► logger.info / warn / error
                            │
                            ▼
                    logger.js (JSON)
                            │
              ┌─────────────┴─────────────┐
              ▼                           ▼
         stdout (консоль)          logs/api.log (LOG_FILE)
              │                           │
              ▼                           ▼
    docker logs / journald          npm run logs:analyze
    GitHub Actions (CI)             AI-промпты + grep/jq
```

**Важно:** фронтенд на **Netlify** не пишет в этот лог — централизованное хранение настроено для **Express API** (`npm run api:start`).

## 1. Структурированные JSON-логи

### 1.1. Модуль `src/server/utils/logger.js`

Каждая запись содержит обязательные поля:

| Поле        | Описание                                          |
| ----------- | ------------------------------------------------- |
| `timestamp` | ISO 8601 (`2026-05-24T10:00:00.000Z`)             |
| `level`     | `error` \| `warn` \| `info` \| `http` \| `debug`  |
| `service`   | Имя сервиса (по умолчанию `stepup-api`)           |
| `message`   | Краткое описание события                          |
| `...`       | Контекст: `method`, `path`, `err`, `stack` и т.д. |

**Пример — info (старт API):**

```json
{
  "timestamp": "2026-05-24T10:00:00.000Z",
  "level": "info",
  "service": "stepup-api",
  "message": "REST API started",
  "url": "http://localhost:3001",
  "database": "postgresql",
  "logFile": "logs/api.log"
}
```

**Пример — http (access log):**

```json
{
  "timestamp": "2026-05-24T10:00:05.123Z",
  "level": "http",
  "service": "stepup-api",
  "message": "HTTP request",
  "method": "GET",
  "path": "/api/health",
  "status": 200,
  "durationMs": 12,
  "ip": "::1"
}
```

**Пример — error (необработанное исключение):**

```json
{
  "timestamp": "2026-05-24T10:03:30.000Z",
  "level": "error",
  "service": "stepup-api",
  "message": "Unhandled error",
  "err": "Connection terminated unexpectedly",
  "stack": "Error: Connection terminated unexpectedly\n    at Pool.query (...)"
}
```

### 1.2. HTTP access log

Раньше использовался **morgan** с текстовой строкой. Теперь `requestLogMiddleware` пишет **структурированный JSON** при завершении каждого запроса (`response.on("finish")`).

Access-логи (`http`) **всегда** записываются, независимо от `LOG_LEVEL`.

## 2. Уровни логирования

| Уровень | Когда используется                            | Пример `message`          |
| ------- | --------------------------------------------- | ------------------------- |
| `error` | 500, необработанные исключения, падение БД    | `Unhandled error`         |
| `warn`  | 401, 403, CORS, невалидный JSON               | `API 401`, `CORS blocked` |
| `info`  | 404, 422, старт/остановка сервера             | `Validation failed`       |
| `http`  | Каждый HTTP-запрос                            | `HTTP request`            |
| `debug` | Отладочная информация (при `LOG_LEVEL=debug`) | —                         |

### Маппинг в `errorMiddleware`

| HTTP-код / ситуация | Уровень | Сообщение в логе      |
| ------------------- | ------- | --------------------- |
| Invalid JSON body   | `warn`  | `Invalid JSON body`   |
| 401 / 403           | `warn`  | `API 401` / `API 403` |
| 422 Validation      | `info`  | `Validation failed`   |
| 404 / 400 / 409     | `info`  | `API 404` и т.д.      |
| 500+ AppError       | `error` | `API 500`             |
| CORS blocked        | `warn`  | `CORS blocked`        |
| Не AppError (crash) | `error` | `Unhandled error`     |

### Переменная `LOG_LEVEL`

В `.env`:

```env
LOG_LEVEL=info
```

Фильтрация: при `info` пишутся `error`, `warn`, `info` (но не `debug`). Уровень `http` для access-логов не фильтруется.

Допустимые значения: `error` | `warn` | `info` | `http` | `debug`.

## 3. Централизованное хранение логов

### 3.1. Файл `logs/api.log` (JSONL)

При заданном `LOG_FILE` каждая запись дублируется в файл — **один центральный источник** для анализа:

```env
LOG_FILE=logs/api.log
LOG_SERVICE=stepup-api
```

| Куда пишется     | Назначение                                   |
| ---------------- | -------------------------------------------- |
| **stdout**       | `docker logs`, systemd journal, терминал     |
| **logs/api.log** | Поиск, `npm run logs:analyze`, загрузка в AI |

Файлы `logs/*.log` добавлены в `.gitignore` — в репозиторий не коммитятся.

### 3.2. Docker / production

Postgres уже в Docker (`infra/docker-compose.yml`). API запускается на хосте; логи доступны через:

```powershell
# stdout при локальном запуске
npm run api:start

# или перенаправление в файл
npm run api:start 2>&1 | Tee-Object -FilePath logs/api.log
```

Для полноценного ELK / Grafana Loki / Datadog нужен отдельный хост API — в рамках MVP достаточно **JSONL-файла + stdout**.

### 3.3. CI/CD (GitHub Actions)

Логи pipeline (lint, test, deploy, smoke test) хранятся **централизованно в GitHub Actions** → каждый workflow run → вкладка job logs. Это мониторинг сборки, не runtime API.

## 4. Анализ логов

### 4.1. Скрипт `scripts/analyze-logs.mjs`

```powershell
# Сводка по всем записям
npm run logs:analyze

# Только ошибки
npm run logs:analyze -- --level error

# Поиск по тексту
npm run logs:analyze -- --search "PostgreSQL"

# Последние 20 записей
npm run logs:analyze -- --last 20
```

**Демо-данные** (без запущенного API):

```powershell
npm run logs:demo
npm run logs:analyze
```

### 4.2. Ручной анализ (PowerShell / jq)

```powershell
# Все error за сегодня
Select-String -Path logs/api.log -Pattern '"level":"error"'

# С jq (если установлен)
Get-Content logs/api.log | jq 'select(.level=="error")'
```

## 5. Промпты для поиска проблем в логах (AI)

Скопируйте фрагмент `logs/api.log` или stdout и используйте промпты ниже в Cursor / ChatGPT.

### 5.1. Общий разбор

```text
Проанализируй JSONL-логи Node.js API (StepUp). Найди:
1) все записи level=error и их probable root cause;
2) повторяющиеся warn за последний период;
3) аномалии по durationMs в http-логах (медленные запросы > 500ms);
4) корреляцию ошибок с path и method.
Выведи таблицу: время | level | message | path | рекомендация.
```

### 5.2. Проблемы с авторизацией

```text
В логах StepUp API найди все записи с message "API 401" или code "UNAUTHORIZED".
Сгруппируй по path. Есть ли brute-force на /api/auth/login?
Предложи, нужен ли rate limit или блокировка IP.
```

### 5.3. Проблемы с базой данных

```text
Найди в JSONL-логах: "PostgreSQL connection failed", "Connection terminated",
"ECONNREFUSED", database disconnected. Определи последовательность событий
и что проверить первым (DATABASE_URL, docker compose, pool limits).
```

### 5.4. Ошибки валидации (422)

```text
Отфильтруй логи с message "Validation failed". Какие path чаще всего?
Какие поля в details повторяются? Это баг клиента или слабая валидация на фронте?
```

### 5.5. CORS и production

```text
Найди "CORS blocked" в логах. С какими origin приходят запросы?
Сверь с API_CORS_ORIGINS и Netlify URL. Дай точное значение для .env.
```

### 5.6. После deploy (корреляция с CI)

```text
Сопоставь время записей в api.log со временем GitHub Actions deploy.
Были ли error/http 5xx сразу после деплоя? Это регрессия или инфраструктура?
```

## 6. Тестирование типичных ошибок

### 6.1. Unit-тесты логгера

Файл: `src/server/utils/logger.test.js`

| Тест                | Проверка                                     |
| ------------------- | -------------------------------------------- |
| Structured JSON     | Поля `timestamp`, `level`, `service`         |
| LOG_LEVEL threshold | `info` скрывает `debug`                      |
| Error + stack       | `err`, `stack` в записи                      |
| `warning` alias     | Пишет как `warn`                             |
| HTTP always logged  | При `LOG_LEVEL=error` http всё равно пишется |

```powershell
npm test -- src/server/utils/logger.test.js
```

### 6.2. Сценарии типичных ошибок

## 7. Переменные окружения

| Переменная         | По умолчанию | Описание                            |
| ------------------ | ------------ | ----------------------------------- |
| `LOG_LEVEL`        | `info`       | Минимальный уровень (кроме `http`)  |
| `LOG_SERVICE`      | `stepup-api` | Поле `service` в каждой записи      |
| `LOG_FILE`         | —            | Путь к JSONL-файлу (`logs/api.log`) |
| `DISABLE_HTTP_LOG` | —            | `1` — отключить access-логи         |

Пример в `.env.example`:

```env
LOG_LEVEL=info
LOG_SERVICE=stepup-api
LOG_FILE=logs/api.log
```

## 11. Ограничения

| Компонент           | Логируется? | Комментарий                                     |
| ------------------- | ----------- | ----------------------------------------------- |
| Express API         | ✅          | stdout + `LOG_FILE`                             |
| Frontend (Netlify)  | ❌          | Только browser console / Metrika                |
| PostgreSQL          | Частично    | Ошибки через `/api/health` и `logger.error`     |
| ELK / Loki / Splunk | ❌ MVP      | Можно подключить позже, читая JSONL или stdout  |
| Ротация логов       | ❌ MVP      | Рекомендация: logrotate или ограничение размера |
