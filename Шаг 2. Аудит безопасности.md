# Шаг 2. Аудит безопасности

Документ фиксирует проведённый аудит безопасности проекта **StepUp Web MVP**: проверку зависимостей, анализ кода по OWASP Top 10, внесённые исправления и рекомендации.

---

## 1. Методология аудита

| Область                  | Инструмент / подход                                                          |
| ------------------------ | ---------------------------------------------------------------------------- |
| Зависимости (production) | `npm audit --omit=dev`                                                       |
| Зависимости (все)        | `npm audit`                                                                  |
| Статический анализ кода  | ESLint + ручной review                                                       |
| OWASP Top 10             | Чеклист по категориям (см. раздел 3)                                         |
| CI                       | Job **Lint & Format** + `npm run audit:prod` в `.github/workflows/ci-cd.yml` |

**Примечание:** `safety` применяется к Python-проектам. Для Node.js используется `npm audit`.


Обновлены **express**, **qs**, **ws** и связанные transitive-зависимости без major-upgrade.

### 2.3. После исправлений

| Команда                | Результат                                                       |
| ---------------------- | --------------------------------------------------------------- |
| `npm audit --omit=dev` | **0 vulnerabilities** (production)                              |
| `npm audit`            | **5 moderate** — только **dev**-цепочка vite → esbuild → vitest |

### 2.4. Оставшийся риск (dev-only)

Уязвимость **esbuild/vite** касается **локального dev-сервера** (`npm run dev`), не production-сборки и не Netlify. Устранение потребовало бы major-upgrade (vite 8.x) с риском поломки тестов.

**Решение:** принят как **допустимый dev-риск**; в CI проверяются только production-зависимости (`npm run audit:prod`).

### 2.5. Новые security-зависимости

| Пакет                  | Назначение                    |
| ---------------------- | ----------------------------- |
| **helmet**             | HTTP security headers для API |
| **express-rate-limit** | Ограничение частоты запросов  |

## 3. OWASP Top 10 — результаты анализа

### A01: Broken Access Control

| Проверка                    | Статус | Комментарий                                      |
| --------------------------- | ------ | ------------------------------------------------ |
| JWT на защищённых маршрутах | OK     | `requireAuth`, `requireRole('admin')`            |
| Заказы только своего userId | OK     | `listByUserId`, `findByOrderNumber(..., userId)` |
| IDOR по orderNumber         | OK     | фильтр `user_id` в SQL                           |

**Исправления:** явный алгоритм JWT `HS256` при sign/verify (защита от alg-confusion).

---

### A02: Cryptographic Failures

| Проверка           | Статус   | Комментарий                       |
| ------------------ | -------- | --------------------------------- |
| Пароли в БД        | OK       | bcrypt (cost 10)                  |
| JWT secret в prod  | OK       | `validateEnv`: минимум 32 символа |
| Dev default secret | Риск dev | только для тестов/in-memory       |

**Исправления:** `jwt.verify(..., { algorithms: ['HS256'] })`, `jwt.sign(..., { algorithm: 'HS256' })`.

---

### A03: Injection (SQL)

| Проверка                      | Статус     | Комментарий                        |
| ----------------------------- | ---------- | ---------------------------------- |
| PostgreSQL-запросы            | OK         | везде параметризация `$1, $2, ...` |
| Конкатенация user input в SQL | Не найдено |                                    |

**Исправления:**

- `isValidOrderNumber()` — формат `SU-\d{6}` перед запросом к БД;
- лимиты длины строк в валидации (снижение DoS через большие payload).

Файлы: `src/server/repositories/postgres/*`, `src/server/validation/ordersValidation.js`.

---

### A03: Injection (XSS)

| Проверка                                | Статус          | Комментарий                          |
| --------------------------------------- | --------------- | ------------------------------------ |
| `innerHTML` / `dangerouslySetInnerHTML` | Не используется |                                      |
| Вывод данных API                        | OK              | везде `textContent`, `createElement` |

**Исправления (defence in depth):**

- CSP и security headers в `netlify.toml` для статического фронта на Netlify.

---

### A04: Insecure Design

| Проверка                 | Статус | Комментарий                               |
| ------------------------ | ------ | ----------------------------------------- |
| Валидация входных данных | OK     | отдельные модули validation               |
| Ошибки 500 в prod        | OK     | generic message при `NODE_ENV=production` |

**Исправления:** лимиты длины полей (`stringLimits.js`), лимит JSON body `100kb`.

| Проверка             | До                            | После                                                |

## 4. Защита от распространённых атак

### 4.1. XSS (Cross-Site Scripting)

| Уровень  | Мера                                                         |
| -------- | ------------------------------------------------------------ |
| Frontend | `textContent` вместо HTML-инъекций                           |
| Netlify  | `Content-Security-Policy`, `X-Content-Type-Options: nosniff` |
| API      | JSON API, не отдаёт HTML                                     |

### 4.2. CSRF (Cross-Site Request Forgery)

| Факт                                                 | Мера                                                |
| ---------------------------------------------------- | --------------------------------------------------- |
| API использует **Authorization: Bearer** (не cookie) | Классический CSRF для cookie **не применим**        |
| CORS whitelist в production                          | `API_CORS_ORIGINS` — браузер блокирует чужие origin |

Дополнительный CSRF-токен не требуется при текущей схeme JWT в заголовке.

### 4.3. SQL Injection

| Мера                  | Реализация                               |
| --------------------- | ---------------------------------------- |
| Parameterized queries | `$1`, `$2` во всех postgres-репозиториях |
| Валидация типов       | `toIntId()`, `parsePositiveInteger()`    |
| Whitelist формата     | `isValidOrderNumber()` для `SU-XXXXXX`   |

### 4.4. DoS / brute-force

| Мера                 | Реализация                                                     |
| -------------------- | -------------------------------------------------------------- |
| Rate limit (общий)   | 300 запросов / 15 мин на IP                                    |
| Rate limit (auth)    | 20 попыток / 15 мин на `/api/auth/login`, `/api/auth/register` |
| Body size limit      | `express.json({ limit: '100kb' })`                             |
| String length limits | email ≤254, password ≤128, текстовые поля ≤200–500             |

### 4.5. Прочее

| Атака                  | Мера                                                  |
| ---------------------- | ----------------------------------------------------- |
| Clickjacking           | `X-Frame-Options: DENY`, CSP `frame-ancestors 'none'` |
| JWT alg:none           | Явный `algorithms: ['HS256']`                         |
| Information disclosure | 500 без stack trace в production                      |

## 7. Рекомендации на будущее

1. **Production API:** задать `API_CORS_ORIGINS=https://regal-sunflower-1a7fff.netlify.app` (и другие домены через запятую).
2. **HTTPS:** Netlify и reverse proxy — только TLS.
3. **Secrets:** ротация `JWT_SECRET`, не коммитить `.env`.
4. **Dev-сервер:** не expose Vite dev server в публичную сеть (риск esbuild GHSA).
5. **Зависимости:** периодически `npm audit fix` и обновление vite/vitest при выходе stable major без уязвимостей.
6. **Backend на Netlify:** API не на Netlify — развернуть Express отдельно с теми же env и CORS.
