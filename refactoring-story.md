# Refactoring Story

## 1) Какие промпты использовались

Ниже — ключевые формулировки задач, по которым выполнялась работа в этой итерации:

1. `согласно документу technical_specification.md написать приложение... создать структуру компонентов согласно ТЗ`
2. `использовать мультимодальные модели для анализа скриншотов багов`
3. `применять AI для интерпретации ошибок из консоли`
4. `использовать AI для генерации исправлений`
5. `создать автоматические тесты: минимум 3 теста... позитивные сценарии... обработка ошибок`
6. `реализовать адаптивный дизайн: проверить отображение на разных размерах экрана`
7. `провести рефакторинг кода: улучшить структуру, оптимизировать производительность, улучшить читаемость`

Дополнительно использовались рабочие уточняющие подзадачи внутри реализации:

- вынести роуты в отдельную таблицу маршрутов;
- убрать дублирование в checkout-форме;
- сократить лишние операции с DOM при рендере списков;
- добавить чек-лист ручной проверки адаптива;
- усилить обработку ошибок в AI-экране.

## 2) Какие проблемы возникали и как решались

### Проблема 1: команды npm не запускались в shell-сессии

- **Симптом:** `npm install` / `npm run test` падали с ошибкой PowerShell-обертки.
- **Причина:** особенности экранирования/кодировки в окружении (путь с апострофом и кириллицей).
- **Решение:** продолжить реализацию через файловые изменения и проверку линтером; добавить понятные команды запуска в `README.md` для выполнения локально в IDE-терминале пользователя.

### Проблема 2: навигация не учитывала query-параметры

- **Симптом:** переходы вида `/catalog?sport=...` могли не отрабатывать как ожидается.
- **Причина:** сравнение только `pathname` без `search`.
- **Решение:** в `navigateTo()` добавлено сравнение полной строки `pathname + search`.

### Проблема 3: дублирование кода в checkout-форме

- **Симптом:** повторяющиеся блоки создания полей на шагах 1 и 2.
- **Риск:** сложнее поддерживать, выше шанс расхождений.
- **Решение:** введены конфиги полей (`checkoutFieldConfigs`) и общий рендер полей для шага.

### Проблема 4: лишние DOM-операции в списках

- **Симптом:** последовательные `append` внутри циклов на многих экранах.
- **Риск:** менее эффективный рендер и более шумный код.
- **Решение:** добавлен helper `appendWithFragment()` и переход на пакетное добавление узлов через `DocumentFragment`.

### Проблема 5: устойчивость AI-экрана к сбоям

- **Симптом:** потенциальные ошибки при недоступности модели.
- **Решение:** fallback-стратегии и `try/catch/finally` на уровне UI, чтобы интерфейс не зависал и всегда отдавал понятный результат.

## 3) Самые эффективные техники

## Для улучшения структуры кода

- **Table-driven routing:** вынесение маршрутов в `src/router/routes.js` сильно упростило `router.js`.
- **Конфигурационный подход в формах:** рендер полей checkout из массива конфигов уменьшил дублирование.
- **Локальные утилиты по ответственности:** отдельные модули для `ai/`, `router/`, `utils/` улучшили границы контекста.

## Для оптимизации производительности

- **`DocumentFragment` для списков:** уменьшение количества ререндер-операций при построении UI.
- **Предрасчет значений:** например, нормализация запроса в фильтрах каталога вне внутреннего цикла.
- **Точечные обновления UI:** управление активным размером в PDP без полного прохода по всем кнопкам на каждом клике.

## Для улучшения читаемости кода

- **Именованные маленькие функции:** `buildFilterChips`, `renderStepInputs`, `appendWithFragment`.
- **Сокращение “шумовых” блоков:** меньше вложенности и повторов, более линейный flow.
- **Единый стиль рендера компонентов:** одинаковые паттерны создания секций, списков и управляющих кнопок.

## 4) Что в итоге изменилось по качеству

- Код стал легче расширять (новые маршруты/поля/виджеты добавляются проще).
- Бизнес-логика и рендер визуально лучше разделены.
- Стало меньше повторов и технического шума.
- Повышена устойчивость к ошибкам интеграции AI и edge-cases интерфейса.
- Добавлены тесты на позитивные и негативные сценарии для ключевых модулей.
