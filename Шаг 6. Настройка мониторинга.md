# Шаг 6. Настройка мониторинга

Документ описыет мониторинг проекта **StepUp Web MVP** с опорой на **Netlify** (production-фронт), health check endpoints backend и автоматические проверки в **GitHub Actions**.

---

## Цель

- Настроить мониторинг доступности приложения на **Netlify**.
- Создать **Health Check endpoints** для API (живость, готовность, полная диагностика).
- Проверять доступность **PostgreSQL** и статус **внешних интеграций** (OAuth).
- Настроить **алерты** при проблемах (уведомления Netlify + failed job в CI).
- Автоматически проверять production после каждого deploy.

---

## Архитектура мониторинга

```text
GitHub Actions (push → deploy)
        │
        ├─► Deploy to Netlify (production-branch: master)
        └─► Smoke test: deploy_url → GET /health.json + GET /
                │
                ▼
Netlify (production SPA)
        │
        ├─► /health.json          — статический health check фронтенда
        ├─► /                     — главная страница (HTTP 200)
        └─► Deploy notifications  — email при failed deploy

Backend (localhost / отдельный хост)
        │
        ├─► GET /api/health/live   — процесс жив (без БД)
        ├─► GET /api/health/ready  — готовность (проверка БД)
        └─► GET /api/health        — полная диагностика (БД + OAuth)
```

**Важно:** Netlify раздаёт **только статику** (`dist/`). Endpoints `/api/health/*` работают на **Express API** (`npm run api:start`), не на Netlify.

## 1. Мониторинг на Netlify

### 1.1. Статический health check — `/health.json`

При каждой сборке генерируется файл `public/health.json`:

```json
{
  "status": "ok",
  "service": "stepup-frontend",
  "timestamp": "2026-05-24T12:00:00.000Z",
  "environment": "build"
}
```

| Компонент           | Файл                                                                 |
| ------------------- | -------------------------------------------------------------------- |
| Генерация при build | `scripts/write-health-json.mjs`                                      |
| Исходник / fallback | `public/health.json`                                                 |
| Команда             | `npm run build` → `node scripts/write-health-json.mjs && vite build` |
| Production URL      | `https://regal-sunflower-1a7fff.netlify.app/health.json`             |

**Проверка в браузере или curl:**

```powershell
curl https://regal-sunflower-1a7fff.netlify.app/health.json
```

Ожидание: `"status": "ok"`, HTTP **200**.

### 1.2. Маршрутизация и заголовки (`netlify.toml`)

**Redirect для `/health.json`** — правило **до** SPA-fallback `/* → index.html`, чтобы статический JSON не подменялся `index.html`:

```toml
[[redirects]]
  from = "/health.json"
  to = "/health.json"
  status = 200
  force = false

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

**Cache-Control** для `/health.json`:

```toml
Cache-Control: no-cache, no-store, must-revalidate
```

Мониторинг всегда получает актуальный ответ, а не закэшированный.

### 1.3. Уведомления Netlify (алерты при проблемах deploy)

Настройка **вручную** в [app.netlify.com](https://app.netlify.com):

1. Откройте сайт **regal-sunflower-1a7fff** (или ваш Site name).
2. **Project configuration → Notifications** (или **Site settings → Build & deploy → Deploy notifications**).
3. Добавьте **Email notifications**:
   - **Deploy failed** — ✅ включить (алерт при ошибке сборки/деploy).
   - **Deploy succeeded** — по желанию.
4. (Опционально) **Slack / webhook** — для команды.

Это **встроенный мониторинг хостинга Netlify**: вы узнаете о проблемах, если production deploy не прошёл.

### 1.4. Дашборд Netlify (ручной мониторинг)

| Раздел Netlify    | Что смотреть                                  |
| ----------------- | --------------------------------------------- |
| **Deploys**       | Статус последнего deploy (Published / Failed) |
| **Deploy log**    | Ошибки сборки `npm run build`                 |
| **Site overview** | Production URL, SSL, last publish time        |

Рекомендация из Шага 1: **Stop builds** в Netlify — deploy только из GitHub Actions, без дублирования.

### 1.5. Production URL

| Параметр          | Значение                                            |
| ----------------- | --------------------------------------------------- |
| Site URL          | `https://regal-sunflower-1a7fff.netlify.app`        |
| Health (frontend) | `/health.json`                                      |
| SPA               | `/`, `/catalog`, `/cart` (redirect на `index.html`) |

Опционально в GitHub Secrets:

| Secret               | Назначение                                                      |
| -------------------- | --------------------------------------------------------------- |
| `NETLIFY_SITE_URL`   | Fallback URL для smoke test, если action не вернул `deploy_url` |
| `NETLIFY_SITE_ID`    | Должен совпадать с Site ID в Netlify UI                         |
| `NETLIFY_AUTH_TOKEN` | Токен для deploy из Actions                                     |

**Важно:** production-ветка в репозитории — **`master`** (не `main`). В CI задано `production-branch: master`.

## 2. Health Check endpoints (Backend API)

### 2.1. Модуль `src/server/health/healthHandlers.js`

| Endpoint                | Назначение                   | Проверка БД | HTTP при ошибке БД |
| ----------------------- | ---------------------------- | ----------- | ------------------ |
| `GET /api/health/live`  | Liveness — процесс отвечает  | Нет         | —                  |
| `GET /api/health/ready` | Readiness — готов к запросам | Да          | **503**            |
| `GET /api/health`       | Полная диагностика           | Да + OAuth  | **503**            |

### 2.2. Пример ответа `GET /api/health`

```json
{
  "status": "ok",
  "service": "stepup-api",
  "timestamp": "2026-05-24T12:00:00.000Z",
  "uptimeSeconds": 3600,
  "check": "health",
  "checks": {
    "database": "connected",
    "oauth": "configured"
  }
}
```

| Поле `checks` | Значения                                        | Как определяется                            |
| ------------- | ----------------------------------------------- | ------------------------------------------- |
| `database`    | `connected` / `disconnected` / `not_configured` | `SELECT 1` в PostgreSQL                     |
| `oauth`       | `configured` / `not_configured`                 | наличие `YANDEX_OAUTH_CLIENT_ID` и `SECRET` |

При `database: "disconnected"` → `"status": "degraded"`, HTTP **503**.

### 2.3. Подключение в `src/server/index.js`

```js
healthCheck: async () => {
  await pool.query("SELECT 1");
},
getExternalServices: () => ({
  oauth: env.yandexOAuth.clientId && env.yandexOAuth.clientSecret
    ? "configured"
    : "not_configured"
})
```

### 2.4. Локальная проверка API

```powershell
cd infra
docker compose up -d
cd ..
npm run api:start
```

```powershell
curl http://localhost:3001/api/health/live
curl http://localhost:3001/api/health/ready
curl http://localhost:3001/api/health
```

Ожидание при работающем Postgres: `"status": "ok"`, `"database": "connected"`.

**Проверка degraded:** остановите Postgres (`docker compose stop`) и повторите `curl /api/health` → HTTP **503**.

## 3. Автоматический мониторинг в CI/CD

После deploy job **Deploy to Netlify** выполняет **smoke test** (`.github/workflows/ci-cd.yml`):

```yaml
- name: Deploy to Netlify
  id: netlify-deploy
  uses: nwtgck/actions-netlify@v3.0
  with:
    publish-dir: ./dist
    production-branch: master # в репозитории ветка master, не main
  env:
    NETLIFY_AUTH_TOKEN: ${{ secrets.NETLIFY_AUTH_TOKEN }}
    NETLIFY_SITE_ID: ${{ secrets.NETLIFY_SITE_ID }}

- name: Smoke test Netlify health endpoints
  run: |
    SITE_URL="${{ steps.netlify-deploy.outputs.deploy_url }}"
    if [ -z "$SITE_URL" ]; then
      SITE_URL="${NETLIFY_SITE_URL:-https://regal-sunflower-1a7fff.netlify.app}"
    fi
    echo "Checking $SITE_URL"
    sleep 10
    curl -fsS "$SITE_URL/health.json" | node -e "..."
    curl -fsS -o /dev/null "$SITE_URL/"
```

| Особенность                 | Зачем                                                          |
| --------------------------- | -------------------------------------------------------------- |
| `id: netlify-deploy`        | Smoke test использует **реальный URL** deploy из output action |
| `production-branch: master` | Production на Netlify обновляется при push в `master`          |
| `sleep 10`                  | Короткая пауза, пока CDN Netlify подхватит новый deploy        |
| Fallback `NETLIFY_SITE_URL` | Если output пустой — проверка по secret или default URL        |

| Событие                    | Алерт                                                 |
| -------------------------- | ----------------------------------------------------- |
| Smoke test failed          | ❌ Job **Deploy to Netlify** красный в GitHub Actions |
| Deploy failed              | ❌ Email от Netlify (если настроено п. 1.3)           |
| Lint / test / build failed | ❌ Workflow CI/CD останавливается до deploy           |

**GitHub → Actions → CI/CD** — главная панель мониторинга pipeline.

### 3.1. Устранение неполадки: smoke test → 404

**Симптом:** deploy зелёный, smoke test падает с `curl: (22) ... 404` на `/health.json`.

**Причины и решение:**

| Причина                                                     | Решение                                                |
| ----------------------------------------------------------- | ------------------------------------------------------ |
| `production-branch: main`, а push в **`master`**            | Исправлено: `production-branch: master`                |
| Smoke test проверяет **чужой URL**, не тот куда ушёл deploy | Использовать `steps.netlify-deploy.outputs.deploy_url` |
| `NETLIFY_SITE_ID` не совпадает с сайтом в Netlify           | Сверить Site ID в Netlify → Settings → General         |
| `health.json` не в `dist/`                                  | `npm run build` должен создавать `dist/health.json`    |

**Коммит с исправлением:** `Fix Netlify deploy: production-branch master, smoke test uses deploy URL.`

## 4. Алерты при проблемах — сводка

| Источник                  | Триггер                       | Куда приходит                           |
| ------------------------- | ----------------------------- | --------------------------------------- |
| **Netlify Notifications** | Deploy failed                 | Email (настройка в UI)                  |
| **GitHub Actions**        | Smoke test / lint / test fail | Email GitHub + красный статус в PR/push |
| **Ручная проверка**       | `/health.json` не открывается | Диагностика: Netlify Deploys, CI logs   |

### Рекомендуемая настройка алертов (чеклист)

```text
□ Netlify → Notifications → Deploy failed → Email
□ GitHub → Watch repository → Actions failures
□ NETLIFY_SITE_ID совпадает с Netlify UI
□ production-branch в CI = master (как в репозитории)
□ (Опционально) Secret NETLIFY_SITE_URL — fallback для smoke test
```

## 5. Как проверить (чеклист для отчёта)

| #   | Действие                                          | Ожидание                                   |
| --- | ------------------------------------------------- | ------------------------------------------ |
| 1   | `curl localhost:3001/api/health` (API + Postgres) | 200, `checks.database: connected`          |
| 2   | `curl localhost:3001/api/health/live`             | 200, `check: live`                         |
| 3   | Остановить Postgres → `curl /api/health`          | **503**, `degraded`                        |
| 4   | `curl https://...netlify.app/health.json`         | 200, `"status":"ok"`                       |
| 5   | Push в `master` → GitHub Actions                  | Deploy + Smoke test ✅                     |
| 6   | Netlify → Deploys                                 | Published                                  |
| 7   | Netlify Notifications                             | Email при failed deploy (тест опционально) |

### Скриншоты для отчёта

1. Ответ `/health.json` на Netlify (браузер или curl).
2. Ответ `/api/health` локально (JSON с `checks`).
3. GitHub Actions — job **Smoke test Netlify health endpoints** ✅.
4. Netlify → **Notifications** (настроенные алерты).
5. Netlify → **Deploys** — успешный production deploy.
